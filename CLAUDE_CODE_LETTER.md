# 给 Claude Code 的一封项目沟通信

Claude，你好。

我和另一个代码助手一起先通读了这个项目，想把当前理解、判断和下一步建议整理给你。目标不是做一份泛泛而谈的 README 补充，而是想请你基于这份上下文，继续判断这个项目下一步最值得投入的方向。

## 1. 这个项目现在到底是什么

它本质上是一个基于 Chrome / Edge Manifest V3 的浏览器扩展，定位更接近：

**当前页媒体捕获 + 流媒体分析 + 下载辅助工具**

而不是：

- 全站爬虫
- DRM 绕过工具
- 自动化批量搬运平台
- 真正意义上的本地转码器

从代码结构看，项目已经有一条相当完整的捕获链：

- `background.js`
  负责 `webRequest` 网络监听、按 tab 维护资源状态、执行下载、分析 `m3u8/mpd`、生成 `ffmpeg` 命令。
- `page-bridge.js`
  注入页面主世界，补抓 `fetch/XHR/media.src/Audio()/createObjectURL()` 这类扩展隔离世界里不容易拿全的线索。
- `content.js`
  扫描 DOM、监听动态节点、渲染页面悬浮面板，并负责页面内 `blob/text` 抓取兜底。
- `popup.js`
  提供弹窗分类列表、下载入口、流媒体分析、复制 `ffmpeg` 命令、抓取开关设置。
- `shared.js`
  承担媒体识别、分类判断、下载模式判断、消息协议与公共工具函数。

所以它不是一个“刚起步的 demo”，而是已经有产品骨架了。真正的问题不是“还能不能再加功能”，而是“应该把它推向哪一种更清晰、更强的产品形态”。

## 2. 我对项目定位的判断

我认为它最有价值的方向，不是继续泛化成“万能下载器”，而是收敛成：

**媒体资源抓取与流媒体复现工作台**

原因很直接：

1. 普通图片/音视频直链下载，本身没有太强壁垒，市面上替代品很多。
2. 这个项目已经有一部分差异化能力，尤其是：
   - 多链路媒体发现
   - `blob:` 资源兜底
   - `m3u8/mpd` 轻量分析
   - `ffmpeg` 命令生成
3. 真正能继续拉开差距的，不是“再多支持几个后缀”，而是“把流媒体地址发现、分析、导出、复现这条链路做深”。

换句话说，我觉得它未来最强的卖点应该是：

> 帮用户更快找到真实媒体入口，并把这个入口变成可复现、可导出、可继续处理的任务。

## 3. 我看到的三条可能路线

### 路线 A：继续做“万能媒体下载器”

方向是继续强化：

- 更多格式识别
- 更多站点兼容
- 更顺手的批量下载
- 更好的 UI 和筛选

这条路能做，但问题是容易同质化，长期会陷入“不断补兼容”的维护泥潭。

### 路线 B：做“流媒体抓取与复现工具”

方向是围绕 `m3u8/mpd` 深挖：

- 更完整的索引解析
- 质量档位/音轨/字幕轨识别
- 请求头、Referer、Cookie、Origin 等复现信息导出
- 输出 `ffmpeg` / `yt-dlp` / `curl` / `aria2` 等命令
- 导出任务上下文，方便在浏览器外继续处理

这是我最推荐的路线，因为它最符合当前代码已经形成的优势。

### 路线 C：做“站点化素材采集器”

方向是引入更强的平台规则：

- 不同网站单独策略
- 智能命名
- 专辑/相册/章节聚合
- 站点模板化导出

这条路线也可行，但维护成本会明显提高，而且会逐步从“通用工具”走向“规则驱动产品”。

## 4. 我的推荐

如果只能选一个主方向，我建议选：

**路线 B，主打流媒体抓取与复现。**

然后保留路线 A 里的基础下载能力，作为配套而不是主卖点。

一句话描述未来产品，可以是：

**让用户在浏览器里发现媒体，在扩展里理解上下文，在浏览器外复现下载。**

## 5. 当前代码里值得继续放大的能力

下面这些能力已经有雏形，适合继续做深：

### 5.1 多来源捕获链

现在的网络监听、主世界桥接、DOM 扫描三条链已经形成互补关系。这个是项目底座，应该继续强化，而不是重写方向。

适合增强的点：

- 给每条资源保留更完整的来源链路
- 标明它来自 `webRequest`、`fetch`、`XHR`、DOM 还是 `blob`
- 给用户更高的“可信度判断”

### 5.2 流媒体分析

这个模块已经不是空白，而是最接近差异化能力的部分。

适合增强的点：

- HLS 主索引和媒体列表的层级展开
- DASH 表示层、音视频轨、字幕轨拆分
- 加密信息识别再细一点
- 直播与点播判断更稳
- 结果可视化，而不是只显示简短摘要

### 5.3 下载策略分流

现在已经区分了：

- 普通直链
- 大文件直下
- `blob:` 回页面抓
- `manifest` 和 `segment` 特殊处理

这说明它不是“一个按钮硬下载到底”。这套策略值得保留，并继续演进成更清晰的“任务系统”。

## 6. 我看到的几个关键短板

这些地方，我认为是下一阶段最值得补的。

### 6.1 资源模型还偏轻

现在很多信息已经收到了，但记录结构还不够强，离“可复现任务”还有差距。

建议补充的字段：

- 请求方法
- 最终跳转地址
- frame 层级
- initiator
- 是否来自缓存
- 请求头快照
- 响应头快照
- Cookie 可用性说明
- 发现链路

### 6.2 资源去重逻辑偏粗

当前按 URL 合并，很容易把不同上下文下的同一 URL 混在一起。

例如这些情况可能需要区分：

- 同 URL，不同 Referer
- 同 URL，不同 frame
- 同 URL，不同 content-type
- 同 URL，对应不同清晰度或不同阶段请求

如果项目往“复现工作台”走，去重键就不能只看 URL。

### 6.3 流媒体解析还属于轻量版

现在的 HLS / DASH 解析对常见场景够用，但还没有进入“工程级可靠”的状态。

例如我会担心这些边界：

- HLS 多音轨、字幕轨、`EXT-X-MEDIA`
- I-frame playlist
- 更复杂的变体关联
- DASH 的 `SegmentList`、`SegmentBase`、多层 `BaseURL`
- 更细的动态直播时间线

### 6.4 浏览器内体验和浏览器外处理之间还没打通

现在已经能复制 `ffmpeg` 命令，但这只是第一步。

更强的形态应该是：

- 导出任务
- 导出命令
- 导出 headers/cookies/context
- 导出为外部工具可直接消费的结构化数据

### 6.5 工程结构还可以再拆

从维护角度看，几个文件已经逐渐变重了，尤其是：

- `background.js`
- `content.js`
- `popup.js`

如果继续做功能，这几个文件会越来越难维护。下一阶段最好尽快把：

- 资源仓库
- 下载执行器
- 流媒体解析器
- 命令生成器
- 站点规则
- UI 渲染

拆成更明确的模块。

## 7. 我建议的产品路线图

下面是我倾向的一版优先级。

### P0：把它从“抓到资源”升级到“理解资源”

先补用户最能感知、也最符合当前方向的能力：

1. 资源详情面板
   - 展示 URL、最终地址、类型、来源链路、frame、Referer、大小、发现时间。
2. 搜索、筛选、排序
   - 按域名、类型、来源、流媒体、时间排序。
3. 批量操作
   - 批量下载、批量复制链接、批量导出 JSON/CSV。
4. 会话保存
   - 保留当前 tab 的采集历史，页面刷新后不直接丢失。

### P1：把流媒体能力做成真正卖点

1. 解析更多 HLS / DASH 结构
2. 列出清晰度、码率、音轨、字幕轨
3. 为每个流媒体资源生成更完整的复现命令
4. 支持导出：
   - `ffmpeg`
   - `yt-dlp`
   - `curl`
   - 原始任务 JSON
5. 识别“这个资源是否大概率无法离站复现”，并说明原因

### P2：把项目变成“任务化工具”

1. 引入任务概念
   - 每个资源不只是一个链接，而是一条任务记录。
2. 任务状态管理
   - 已发现、已分析、已导出、已下载、失败。
3. 历史记录页
   - 可以回看最近分析过的流媒体任务。
4. 收藏与标签
   - 方便处理重复站点和重复流程。

### P3：决定要不要走“扩展 + 本地助手”双端路线

如果后面真想更强，可以考虑增加一个可选本地 companion：

- 浏览器扩展只负责抓取上下文和导出任务
- 本地服务负责实际执行 `ffmpeg`、任务队列、合并结果、重试、日志

但这个不建议太早做。除非已经确认产品目标就是“从嗅探到最终文件落地”。

## 8. 我建议优先做的功能清单

如果只做第一轮增强，我会选下面这些：

1. 资源详情抽屉
2. 批量导出 JSON / `ffmpeg` / 链接列表
3. 更强的流媒体解析结果展示
4. 更合理的资源唯一键与去重逻辑
5. 采集会话持久化
6. 基于站点的命名规则和规则扩展点

这是因为这几项同时满足三件事：

- 用户能立刻感知
- 和当前项目方向一致
- 不需要立刻把架构推翻重来

## 9. 我对工程层的建议

如果你准备继续协助这个项目，我建议你重点盯这几块：

### 9.1 先补测试夹具

至少给 HLS / DASH 解析准备 fixture：

- 标准 VOD m3u8
- 多变体 m3u8
- 带音轨字幕的 HLS
- 动态 MPD
- 带 `SegmentTemplate` 的 MPD
- 带 `SegmentList` 的 MPD

否则后面每加一个解析能力，都容易回归破坏。

### 9.2 把解析逻辑从后台脚本里拆出来

流媒体解析完全值得单独模块化，否则 `background.js` 会继续膨胀。

### 9.3 给站点策略做成可扩展结构

现在已经有 `PLATFORM_STRATEGIES` 的苗头。这个地方值得发展成：

- host matcher
- ignore 规则
- extra media ext
- 文件名规则
- 资源过滤器
- 特殊导出行为

但我建议先做“规则骨架”，不要一开始就塞很多站点特判。

## 11. 我的最终结论

我现在的结论很明确：

这个项目最值得做强的，不是“下载”本身，而是：

**发现真实媒体资源 -> 解释它的上下文 -> 把它导出为可复现任务**

如果沿着这条线走，这个项目会越来越像一个专业工具。

如果沿着“什么都想下载、什么站都要兼容”去走，它很可能会变成一个长期维护压力很大的杂项集合。

你如果继续接手判断，我希望你不要只补“还能加什么功能”，而是继续帮这个项目收敛定位，确保每一步增强都在同一个方向上累积势能。
